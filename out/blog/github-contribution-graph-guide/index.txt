b:T5db2,<h1>A Guide to Display Your GitHub Contribution Graph on Your Personal Website</h1>
<p>This blog post will walk you through how to create a responsive GitHub contribution graph on your website, inspired by the implementation on this portfolio.</p>
<p>We'll cover how to achieve two key layout goals:</p>
<ul>
<li>A full-width, immersive view on desktop screens.</li>
<li>A contained, horizontally scrollable view on mobile devices to maintain a clean layout.</li>
</ul>
<p>You can see the final implementation in the <code>src/components/github-contributions.tsx</code> file of this project.</p>
<h2>Step 1: Fetching Your Contribution Data with the GitHub GraphQL API</h2>
<p>The first step is to get your contribution data from GitHub. While you could scrape your profile page, a much more robust and reliable method is to use the GitHub GraphQL API. This provides a structured way to query for the exact data you need.</p>
<p>In <code>src/components/github-contributions.tsx</code>, we use an async function to send a POST request to <code>https://api.github.com/graphql</code>. The query is structured to fetch the contribution calendar for the past year.</p>
<p>Here's the GraphQL query used in the component:</p>

<p>To authenticate your request, you'll need a Personal Access Token (PAT) from GitHub with the <code>read:user</code> scope. This token is then included in the Authorization header of your request.</p>
<p><strong>Important</strong>: Never expose your PAT on the client-side. In this project, it's stored as an environment variable (<code>NEXT_PUBLIC_GITHUB_TOKEN</code>) and used during the build process to fetch the data.</p>
<p>The fetched data provides an array of weeks, with each week containing an array of <code>contributionDays</code>. Each day object includes the date, <code>contributionCount</code>, and the color that GitHub uses for that day's activity level.</p>

<h2>Step 3: Implementing a Responsive Layout</h2>
<p>This is where we address the two key layout goals. The component uses a responsive design approach to provide an optimal viewing experience on both desktop and mobile.</p>

<p>Here's a breakdown of the mobile layout:</p>
<ul>
<li><code>md:hidden</code>: This ensures the container is only visible on screens smaller than the <code>md</code> breakpoint.</li>
<li><code>overflow-x-auto</code>: This is the key. It tells the browser to create a horizontal scrollbar if the content inside (<code>renderContributionGraph</code>) is wider than the container.</li>
<li><code>min-w-[600px]</code>: We set a minimum width on the inner container to ensure that the graph doesn't get squished, even on very narrow screens. This forces the <code>overflow-x-auto</code> to activate and create the scrollbar.</li>
</ul>
<p>This approach provides a clean and user-friendly experience on all devices, presenting the full graph on desktop while keeping it neatly contained on mobile.</p>

The responsive design ensures that your contribution graph looks great and functions well on any device, from mobile phones to large desktop monitors.</p>
<p>The key takeaways are:</p>
<ol>
<li>Use the GitHub GraphQL API for reliable data fetching</li>
<li>Implement responsive design with different layouts for desktop and mobile</li>
<li>Use horizontal scrolling on mobile to maintain the full graph visibility</li>
<li>Always keep your API tokens secure and never expose them on the client-side</li>
